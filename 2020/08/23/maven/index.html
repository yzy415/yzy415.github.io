Build lifecycles

基本要素

  1. build lifecycle: 三个内置lifecycle，分别为：default, clean, 和site; default负责项目的部署，clean负责项目的清理，site负责文档的生成
  2. build phases: build lifecycles的不同阶段，使用mvn XXX时无需主动声明phases所属的lifecycles。执行某一phase时，会依照顺序把该life cycle前的所有phases执行完。phase对应了不同的goals，打包方式不同，每个phase所执行的goals也不同。使用连字符连起来的phase一半不会直接调用，如intergration test，如果直接调用，调用后，中途使用过得东西不会自动关闭，环境仍处于hanging state
  3. Plugin goals: build phases由多个plugin goals组成，每个goal相当于一个任务，没有和phases绑定的goals可以直接调用，如果没有goal和phase绑定，则该phase不会运行。

设置phase

向phase中添加goals

  1. 修改packaging，通过修改<packaging>来修改打包方式，不同的打包方式下的phase会自动对应到不同的goals。
  2. 配置plugin，plugin是向maven提供goals的aritifacts(通常为jar的一种文件格式，用于部署到maven repositories上)，在pom文件的插件部分，可以声明把该插件的goals绑定到那个phase上，绑定后该goals会自动添加到对应的phase上

POM

  1. super pom: 如非特别声明，所有的pom都间接继承了super pom。

  2. minimal pom:

  3. 项目可以使用<parent>来声明父类pom，groupId、artifactId、version三者共同确定了一个父类artifact

  4. 项目可以使用<module>来把其他模块合并到本项目中，对该项目进行的指令不仅对对该项目作用，对其模块也会进行同样的作用。

Profiles

可以使用profile来定制POM。

profile可以出现在pom.xml、Maven设置、用户设置、以及同目录下的profiles.xml。

可以在命令行中直接通过-P来激活要使用的profiles，可以再Maven settinng中设置默认激活的profiles，也可以在profiles文件中设置条件，运行时检测环境是否符合条件来自动激活，也可以使用-P来失活某个profile。

可以根据激活条件来命名profile，如env-dev profile的激活条件为env=dev

对POM的那部分起作用

取决于profiles在哪里。

  1. 在pom.xml外的，只能声明repositories(存放aritfacts和依赖的地方)和properties(用于定义键值)，不会影响项目的产出结果，保证了项目的可移植性
  2. 在pom.xml里的，可以修改一堆，会直接影响到项目的产出和结果
